# 旅游足迹第二版设计文档

## 目标

在第一版全屏地图旅游足迹页面的基础上，增强单个地点的详情表达能力。

第二版不改变页面整体布局，仍以 Leaflet 全屏地图为核心。升级点集中在地点详情：每个地点除名称和坐标外，新增日期、备注以及本地多图数据，并在 marker 的 popup 中展示横向可滚动的缩略图条。

## 范围

本次只实现以下内容：

- 保持全屏地图布局不变
- 将地点数据扩展为包含 `date`、`note`、`images` 的结构
- 图片资源使用 `public/` 目录下的本地静态路径
- 在 popup 中展示地点名称、日期、备注和横向滚动缩略图条
- 将 popup 内容生成逻辑拆分为独立模块
- 为 popup 内容增加自动化测试
- 保持现有地图初始视野逻辑不变

本次不包含以下内容：

- 独立详情面板或侧边栏
- 图片大图预览、灯箱、轮播
- 图片加载失败占位图
- 地点在线编辑
- 路线连线
- 筛选、分类、搜索
- 后端服务或数据库

## 技术选择

### 方案对比

1. 方案 A：`public/` 静态图片 + `images` 数组 + popup 横向滚动缩略图条
2. 方案 B：`public/` 静态图片 + `images` 数组 + popup 只展示一张图
3. 方案 C：`public/` 静态图片 + `images` 数组 + popup 图条 + 大图预览

### 选定方案

采用方案 A。

原因：

- 最符合当前需求：本地图片、多图、popup 内横向浏览
- 与 Vite `public/` 静态资源模型兼容，路径管理简单
- 在不改变页面结构的前提下，能显著增强地点表达能力
- 避免引入灯箱预览等超出第二版范围的复杂交互

## 页面结构

页面继续使用单屏全屏地图布局。

- 页面主体仍是 Leaflet 地图容器
- 不新增侧边栏、详情面板或额外页面区域
- 所有地点信息仍通过 marker popup 呈现

这保证第二版仍然聚焦于“地图上的旅游足迹”，而不是演变成新的页面结构重构。

## 文件划分

建议第二版按以下职责边界演进：

### `public/images/`

- 存放地点相关的本地图片资源
- 图片通过静态路径在数据中引用，例如 `/images/beijing-1.jpg`

### `src/places.js`

- 继续作为地点数据源
- 将地点结构从 `{ name, lat, lng }` 扩展为：

```js
{
  name,
  lat,
  lng,
  date,
  note,
  images
}
```

- 其中 `images` 是本地图片路径数组，例如：

```js
images: ['/images/beijing-1.jpg', '/images/beijing-2.jpg']
```

### `src/popupContent.js`

- 新增纯函数模块
- 输入单个地点对象
- 输出 popup 所需的 HTML 字符串
- 负责组织标题、日期、备注和图片条结构

### `src/main.js`

- 继续负责地图初始化与 marker 渲染
- 绑定 popup 时调用 `popupContent.js`
- 不再在入口文件中拼接复杂 popup HTML

### `src/style.css`

- 保留原有地图全屏样式
- 新增 popup 内容排版和图片条样式
- 控制 popup 宽度、文本间距、缩略图尺寸和横向滚动行为

### `src/popupContent.test.js`

- 为 popup 生成逻辑提供纯函数测试
- 验证不同字段组合下的输出结构是否符合预期

## 数据模型

第二版地点数据建议结构如下：

```js
{
  name: '北京',
  lat: 39.9042,
  lng: 116.4074,
  date: '2025-10-01',
  note: '故宫和景山的秋天很好看。',
  images: ['/images/beijing-1.jpg', '/images/beijing-2.jpg']
}
```

字段说明：

- `name`：地点名称
- `lat`：纬度
- `lng`：经度
- `date`：旅行日期字符串
- `note`：地点备注
- `images`：本地静态图片路径数组

数据层默认允许 `date`、`note`、`images` 缺失，但渲染层必须安全处理。

## 数据流

第二版页面加载后的数据流如下：

1. `src/main.js` 创建 Leaflet 地图实例
2. 加载底图瓦片图层
3. 从 `src/places.js` 读取地点数组
4. 遍历地点数组并创建 marker
5. 对每个地点调用 `src/popupContent.js` 生成 popup HTML
6. 将 popup 绑定到 marker
7. 使用现有 `src/mapView.js` 逻辑决定初始视野

这保证“地图行为”和“详情内容生成”分离，便于后续继续增强 popup。

## 交互行为

第二版保持交互克制，只增强 popup 内容：

- 页面加载即显示地图
- 所有地点继续显示为 marker
- 点击 marker 后 popup 展示：
  - 地点名称
  - 日期
  - 备注
  - 横向可滚动缩略图条
- 地图初始视野策略不变：
  - 多点：`fitBounds`
  - 单点：`setView`
  - 空数据：默认视野

## 边界场景处理

### `images` 为空或缺失

- 不渲染图片区
- popup 仍展示名称、日期、备注中的可用部分

### `date` 缺失

- 不渲染日期块
- 不输出空标签或占位文案

### `note` 缺失

- 不渲染备注块
- 不输出空标签或占位文案

### 只有一张图片

- 仍使用相同的图片条结构
- 只是不会出现实际滚动行为

### 多张图片

- 在 popup 内显示横向滚动缩略图条
- 每张图片尺寸固定在合理范围内
- popup 保持可控宽度，不因图片数量过多而失控

### 图片路径错误

- 第二版不增加额外错误兜底或占位图
- 默认按本地静态资源路径有效来设计
- 样式上限制容器和图片尺寸，避免布局被异常撑坏

## 样式策略

popup 样式目标是信息更丰富，但仍然轻量：

- popup 主体宽度固定在适合文本与缩略图共存的范围
- 标题、日期、备注按垂直层级排布
- 图片区域使用横向滚动容器
- 图片缩略图尺寸统一，支持圆角和间距
- 内容允许自然换行，不使用复杂响应式排版系统

## 验收标准

第二版完成时应满足：

1. `src/places.js` 支持 `date`、`note`、`images` 字段
2. `images` 使用 `public/` 下的本地静态路径
3. 地图仍能正常显示 marker
4. 点击 marker 后 popup 能显示地点名称、日期、备注和横向滚动缩略图条
5. `images` 为空时，popup 不显示图片区，但布局正常
6. `date` 或 `note` 缺失时，不出现 `undefined` 或空标签
7. 现有初始视野逻辑不被破坏
8. `npm test` 和 `npm run build` 均通过

## 测试策略

第二版采用“纯函数自动化测试 + 浏览器手动验证”的组合：

### 自动化测试

- 保留 `src/mapView.test.js`
- 新增 `src/popupContent.test.js`，覆盖：
  - 完整数据时输出标题、日期、备注、图片列表
  - `images` 为空时不输出图片区
  - `date` 缺失时不输出日期块
  - `note` 缺失时不输出备注块

### 手动验证

- 启动 dev server
- 打开页面并点击不同 marker
- 检查 popup 中多张本地图片是否显示
- 检查图片条是否可横向滚动
- 检查缺字段地点的 popup 是否仍然整洁
- 再次确认地图初始视野行为未回归

## 后续扩展边界

本设计为未来扩展保留清晰边界：

- 若需要独立详情面板，可在保持 `popupContent.js` 独立的前提下复用同一份地点数据
- 若需要图片预览或灯箱，可在当前图片条基础上继续增强交互
- 若需要支持远程数据源，可替换 `src/places.js` 的数据来源而不推翻 popup 结构模块

第二版仍然坚持 YAGNI，只解决“地点详情增强 + 本地多图展示”这一层目标。
